## 2. 既存資料からの確定事項

### 2-1. ROM / 表データ

- 対象ROMサイズは 2,097,152 bytes と整理済み。
- `0x41A10..` は 8バイト固定長の条件分岐表本命。
- `0x41AC0` は単独表ではなく、`0x41A10` から続く大きな表の中腹。
- `0x442BF` 付近は道具表本命、`0x449D0` 付近は装備表本命として扱ってきた。
- 後続の schema 基準解析では、道具表 base は `0x442AC` 寄り、装備表 base は `0x449C3` 寄りに修正されている。

### 2-2. 表/表示系

- `82:FD67` は汎用 field 抽出器。
- `4F` opcode は会話分岐ではなく、UI/施設画面で現在レコードから field を抜く命令として扱うのが安全。
- selector は暫定的に以下の理解。
  - selector 1: 商品系
  - selector 2: 装備系 / 特殊商品系
  - selector 3: キャラ系
  - selector 0: 一覧/基本表示系
- `7E:718B..` は最終描画先ではなく、表示コード staging / operand stack 寄り。

### 2-3. 2026-03-18時点の進展

- `0x41A10 -> 39850/39993` は target-side の施設/イベント blob 系。
- `1D8A -> BDE7 -> C379/C931` は bank82 内の local rule / membership engine 系。
- `19D5` は施設種別そのものではなく choice layer とみるのが自然。
- `ledger (1B15/1B25/1B35) -> bucket (BCA5/BCE1/BD06) -> choice (1D77/1D87)` の中間層が見えている。
- 会話本文そのものの一括抽出は未完成。

## 3. 今回スレッド vol014 で扱った主テーマ

ユーザー指定により、イベント系解析を避け、13目標のうち伸びそうなものを選ぶ方針。

今回スレッド内では主に以下を扱った。

1. GitHub側の現在構成確認
2. 自動タスク設定確認
3. 非イベント系として Goal 7「武器特殊能力サブシステム」を優先候補に設定
4. 武器特殊能力について以下の仮説を段階的に整理
   - 装備表の一部フィールドが特殊カテゴリIDとして使われる可能性
   - 戦闘中に武器特殊IDをCMP連鎖で処理する可能性
   - 特殊処理は即時ダメージ補正と遅延状態処理に分かれる可能性
   - `7E:30xx` を戦闘中の一時状態バッファとして使う可能性
   - 後段で `7E:20xx` 系の恒常状態へ転送する可能性

ただし、これらの武器特殊関連は、このスレッド内でROM/逆アセンブル上のアドレス確定まで十分に裏取りできていないため、**確定資料ではなく要検証のワーキング仮説**として保存する。

過去の自動調査出力では、以下のような内容を「進展」として記述したが、現時点ではアドレス・命令列・ファイル上の根拠が不足している。

1. 逆アセンブル上の実アドレス

3. 装備ID / 武器IDとの参照関係
4. 実行時WRAMトレース
5. 戦闘ログまたはLuaログ

## 5. 次に攻めるべき箇所

イベント系を避ける場合、次は以下の順が安全。

### 優先1: Goal 7 武器特殊能力サブシステムの実証

- `data/weapon_special/` がGitHub側に存在する場合は、その中身を最優先で読む。
- 装備表の `+00..02` 特殊フラグ、`+0E` 装備スロット、`+0F` 能力上昇値を再確認。
- 戦闘処理側で装備ID、武器ID、装備表base、特殊フラグを読む箇所を検索。
- 具体的なCMP/BEQ連鎖、BIT分岐、WRAM書込をアドレス付きで確定する。

### 優先2: Goal 3 WRAM中間構造体

- 既存引継ぎでは `7E:201E..2030` と `7E:3004..303C` が未完成。
- `7E:3004 / 300E / 3014 / 301D / 301E / 301F / 3020 / 302B / 302C / 3030 / 3036 / 303B / 303C / 303E` はD系opcodeの宛先例として重要。
- 武器特殊の仮説と絡める場合は、戦闘用WRAMなのかUI中間構造なのかを必ず切り分ける。

### 優先3: Goal 13 NPC大量表示時の処理軽減

- 既存GitHubでは movement patch / depth reorder skip v2 がある。
- ここは実機/エミュレータ目視検証が必要。
- 静的解析だけで伸ばすより、Luaログ・パッチ検証とセットで進める方がよい。
