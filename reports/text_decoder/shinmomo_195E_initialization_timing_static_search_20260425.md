# 新桃太郎伝説 `$195E` 初期化タイミング追跡メモ  
作成日: 2026-04-25

## 1. 目的

`C4:C8AA` の candidate scan で使われる `$195E` について、  
初期化タイミングを WRAM範囲クリア / 場面切替初期化ルーチンから探した。

---

## 2. 最大の結論

現時点の静的解析では、`$195E` を直接ゼロ初期化する命令は見つからない。

確認済みの直接参照は以下のみ。

| 種別 | address | 命令 | 意味 |
|---|---|---|---|
| read | `C4:C89B` | `LDA $195E` | all candidates consumed check |
| read | `C4:C8AC` | `LDA $195E` | feederで既処理candidateをskip |
| write | `C4:C87A` | `TSB $195E` | 該当candidate bitをset |
| read | `C4:C911` | `BIT $195E` | candidate未消費check |

見つからなかったもの。

```text
STZ $195E
STA $195E
STZ $195E,X
STA $195E,X
```

したがって、`$195E` は専用の「毎回ここで初期化される変数」というより、以下のどちらかで初期化される可能性が高い。

1. 新規場面/イベント開始時の **WRAMブロック汎用クリア** に巻き込まれる  
2. セーブ/ロード/マップ初期化時に **ゼロ状態で始まり、その後はイベント中に蓄積** される  

---

## 3. `$195E` の三点セット

### 3-1. set: `C4:C86E`

```asm
C4:C86E  JSR $C880
C4:C871  BCC $C876
C4:C873  JMP $C366
C4:C876  LDA $84:C8F4,X
C4:C87A  TSB $195E
C4:C87D  JMP $C357
```

候補 `0x10..0x13` / type `1` に該当する場合、対応bitを `$195E` に立てる。

```text
C4:C8F0 = 10 11 12 13
C4:C8F4 = 01 02 04 08
```

### 3-2. all done: `C4:C89B`

```asm
C4:C89B  LDA $195E
C4:C89E  AND #$0F
C4:C8A0  CMP #$0F
```

lower 4bit がすべて立っていれば、candidate `0x10..0x13` は全消費済み。

### 3-3. unconsumed check: `C4:C8F8`

```asm
C4:C90D  LDA $84:C8F4,X
C4:C911  BIT $195E
C4:C914  BEQ $C919
C4:C916  JMP $C366
```

該当bitが立っていれば失敗。  
つまり「このcandidateはまだ未消費か？」を見るpredicate。

---

## 4. WRAM範囲クリア候補の確認

`$195E` を巻き込み得る indexed clear / indexed store を探した。

静的に信頼できるコード側で確認できた主な範囲クリアは以下。

| routine | 内容 | `$195E`を含むか |
|---|---|---|
| `C5:9505` | `STZ $1931,X`, `X<4` | 含まない |
| `C5:97F9` | `STZ $193D,X`, `X<0x15` | 含まない。終端は `$1951` |
| `C5:B298` | `STZ $1952` | 含まない |
| `C3:D203` 系 | `$196D..$1970` 近辺の初期化 | 含まない |
| `C4:C357/C366` | `$1957` success/failure result | 含まない |
| `C4:C85A/C867/C8D6` | `$1958` current candidate | 含まない |
| `C4:C81E` | `$195F` 系 | 含まない |

`$195E` を明示的に含む範囲クリアは見つからない。

## 5. いまの仮説

### 仮説A: `$195E` はイベント内で保持される消費済みmask

`$195E` は candidate `0x10..0x13` の消費済みbitなので、  
イベントの途中ではリセットされず、候補を使うたびにbitが増えていく。

この場合、リセットされるのは以下のタイミングが自然。

- マップ遷移
- イベント開始/終了
- script VM context reset
- new game/load
- WRAM `$1900..$19FF` 系作業領域一括初期化

### 仮説B: 専用初期化なしで、ゼロ初期WRAMに依存

ROM内で専用clearが見つからないため、  
最初にワークRAMがゼロ化された状態から始まり、以降は該当イベント中だけ使う可能性がある。

### 仮説C: 保存対象/復帰対象の一部

`$195E` が場面をまたいで残る場合、candidate消費状態として保存/復帰されている可能性もある。  
ただし現時点では保存ルーチンとの接続は未確認。

---

## 6. Goal 13への意味

`$195E` が「候補消費済みmask」であることはかなり固い。  
ただし初期化タイミングが未確定なので、Goal 13は今回 **96%据え置き** が妥当。

理由:

```text
分かった:
  candidate scan中のset/test/all-doneは確定寄り

未確定:
  いつ$195Eが0へ戻るか
```

もし初期化タイミングが確定すれば、Goal 13は 97〜98% へ上げられる。

---

## 7. runtime確認が必要なwatchpoint

静的解析だけでは「汎用WRAMクリアに巻き込まれるか」を完全には確認しにくい。  
以下のwrite watchpointが有効。

```text
watch write 7E:195E
watch write 7E:1957..7E:1960
watch write 7E:1900..7E:19FF  # 可能なら範囲
```

期待ログ:

| タイミング | 期待されるwrite |
|---|---|
| candidate消費時 | `C4:C87A TSB $195E` |
| マップ/イベント初期化時 | 未知の汎用clearが `$195E=00` |
| candidate scan中 | `$195E` のreadのみ |

---

## 8. 次に攻めるべき箇所

1. script VM / scene reset で `$1900..$19FF` をまとめて初期化するルーチンを探す  
2. `$195E` がセーブ・イベント状態に含まれるか確認する  
3. `C4:C8AA` を呼ぶscript opcode文脈から、candidate scanの発火タイミングを確定する  
4. runtime watchpointで `$195E=00` へのwrite元を実測する  

---

## 9. ROM修正観点

`$195E` 初期化が未確定なため、ここを直接いじるROM修正はまだ危険。

危険な修正:

- `$195E` を毎回0にする
  - candidate消費済み管理を壊し、同じ候補が何度も出る可能性
- `$195E` を常に0x0Fにする
  - candidate scanが即all-done扱いになり、NPC/event候補が消える可能性

現時点の処理改善パッチとしては、引き続き `C1:9474` の通常actor cap変更の方が安全。
