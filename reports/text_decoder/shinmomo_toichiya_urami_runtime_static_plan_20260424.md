# 新桃太郎伝説 追加解析メモ
## といちや説明 / 30000両条件blob / 怨みの洞窟名前入力 hook 方針

作成日: 2026-04-24
対象ROM: `Shin Momotarou Densetsu (J) (1).smc`

## 0. 今回の結論

今回指定された3点について、静的解析で以下まで進めた。

1. `0x082E20..0x083035` は **といちや / 十一屋 / 預かり / 引き出し / 金庫説明** の行単位候補として、かなり確定寄りに上げられる。
2. `0x39839` 近辺の `30 75` は `0x7530 = 30000` と読めるが、直後に `30 76` もあるため、**単純な雇用費即値そのものと断定するのは危険**。現時点では、**といちや加入/預け金条件を扱う target-side script/blob の条件引数** と見るのが最も安全。
3. `0x09BF28..0x09BF5A` は怨みの洞窟の名前入力プロンプト後半だが、静的復元だけでは限界。`$12B4` lineID、`$12B2` charCode、`$12A0..12BF`、`$1180..119F`、名前保存候補RAMを同時にログする runtime hook が必要。

この環境ではエミュレータ実行ログを取得できないため、**runtime lineID と RAM 差分の実測値は未取得**。ただし、取得用 hook と検証観点はここに確定する。

## 2. `0x39839 / 0x39850 / 0x39993` 30000両条件blob 再検証

### 2-1. 周辺バイト列

`0x39835` 付近には、9バイト単位に見える小レコードがある。

```text
0x039832: 22 56 F0 48 00 3E 42 30 75
0x03983B: 00 70 F0 48 00 3E 42 30 76
0x039844: 00 22 E8 38 05 01 01 32 03
0x03984D: 01 9A F0 38 05 01 02 32 10
```

注目点:

```text
0x039839: 30 75  ; little endian 0x7530 = 30000
0x039842: 30 76  ; little endian 0x7630 = 30256
```

`0x7530 = 30000` は、といちやの「3万両」条件と強く整合する。
ただし、直後に `0x7630` もあるため、**この2バイトだけを雇用費そのものと決め打ちするのは危険**。

### 2-2. 41A10条件表からの接続

`0x41A10` 8バイト条件表では、以下の2レコードが `0x398xx/0x399xx` に接続する。

| record | ROM offset | record bytes | target | 解釈 |
|---:|---|---|---|---|
| 68 | `0x041C30` | `74 01 03 02 05 04 50 98` | `0x39850` | A系 target-side blob 2件目 |
| 107 | `0x041D68` | `00 05 03 02 02 02 93 99` | `0x39993` | B系 target-side blob 内部ラベル |

### 2-3. 現時点の分離結論

| 候補 | 評価 | 理由 |
|---|---:|---|
| `0x7530` = といちや雇用費 30000両 | 中 | 金額は一致。ただし transaction core ではなく target-side blob内の条件引数に見える。 |
| `0x7530` = 預け金30000超え判定しきい値 | 高 | `0x41A10 -> 0x39850/0x39993` の条件分岐target近傍にあるため、条件判定側の引数として自然。 |
| `0x7530` = 表示用の数字だけ | 低 | 近傍が9バイトscript/blob列なので、単なる表示文より制御資材寄り。 |

したがって現時点では、

> `0x39839` の `0x7530` は「といちや加入/預け金条件のしきい値」候補を第一候補、
> 実際に30000両を支払う雇用費は別途 transaction helper 側で検証する。

と分離する。

### 2-4. 次に必要な実行時検証

以下を同時ログする。

- 所持金本体級: `$1936` 周辺
- 今回取引額ワーク: `$1969/$196A`
- amount helper: `$1E/$1F`
- ledger: `$1B15/$1B25/$1B35`
- choice/current: `$1986/$198D/$19D5/$19E5`

といちや加入前後で、

1. 預け金が30000以下
2. 預け金が30001以上
3. 雇用実行直前
4. 雇用実行直後

の4点を比較すれば、`0x7530` がしきい値か支払い額か分離できる。

## 3. `0x09BF28..0x09BF5A` 怨みの洞窟 名前入力3本

### 3-1. 静的復元の結論

`0x09BF28..0x09BF5A` は、静的に読むと制御/差し込み/入力呼び出しが混ざり崩れる。
ただし、文脈上は以下でほぼ固定できる。

```text
いちばん 嫌いな人の 名前を お聞かせください！
続いて 2番目に 嫌いな人の 名前を お聞かせください！
最後に 3番目に 嫌いな人の 名前を お聞かせください！
```

### 3-2. runtime hook が必要な理由

この帯は、普通の固定本文ではなく、少なくとも次の要素が混在している。

- 順位表示: 1番目 / 2番目 / 3番目
- 名前入力UI呼び出し
- 入力結果保存
- 後続の敵名/表示名への差し込み
- 文字列stream制御

そのため、静的デコードだけでは `R`, `{21}`, `{40}` 相当の崩れが出る。

### 3-3. 取得すべきRAM窓

既存hookの `$12A0..12BF`、`$1180..119F` に加えて、名前保存候補として以下を追加する。

```lua
TRACE_WINDOW_NAME_A = true -- 0x1200..0x127F
TRACE_WINDOW_NAME_B = true -- 0x1280..0x12FF
TRACE_WINDOW_WORK_D = true -- 0x1D00..0x1FFF
TRACE_WINDOW_UI     = true -- 0x1900..0x19FF
```

### 3-4. 追加hook案

既存の `buildContextRecord()` に以下を追加する。

```lua
if TRACE_WINDOW_NAME_A then
  rec[#rec+1] = "<br><b>RAM[1200..127F]</b>=" .. readRangeHex(0x1200, 0x127F)
end
if TRACE_WINDOW_NAME_B then
  rec[#rec+1] = "<br><b>RAM[1280..12FF]</b>=" .. readRangeHex(0x1280, 0x12FF)
end
if TRACE_WINDOW_WORK_D then
  rec[#rec+1] = "<br><b>RAM[1D00..1FFF]</b>=" .. readRangeHex(0x1D00, 0x1FFF)
end
if TRACE_WINDOW_UI then
  rec[#rec+1] = "<br><b>RAM[1900..19FF]</b>=" .. readRangeHex(0x1900, 0x19FF)
end
```

### 3-5. 実行手順

1. 怨みの洞窟入口直前の savestate を作る。
2. `marker:urami_before_prompt` を送る。
3. 警告文から名前入力1本目まで進める。
4. 1人目に `ああああ`、2人目に `いいいい`、3人目に `うううう` を入力。
5. 各入力直前/直後に marker を打つ。

推奨marker:

```text
marker:urami_before_prompt
marker:urami_prompt_1_before_input
marker:urami_prompt_1_after_input_aaaa
marker:urami_prompt_2_before_input
marker:urami_prompt_2_after_input_iiii
marker:urami_prompt_3_before_input
marker:urami_prompt_3_after_input_uuuu
marker:urami_enemy_name_display
```

6. `84:B5E3` のline flushログで `$12B4` lineID と本文を取る。
7. 入力直後のRAM差分から、`ああああ / いいいい / うううう` が保存されたアドレスを特定する。
8. 敵名表示時に同じRAMが参照されるか確認する。
