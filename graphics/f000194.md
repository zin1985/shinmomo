# 新桃太郎伝説解析 vol016 グラフィック・マップチップ復元 初回引継ぎ（2026-05-01）

## 0. 今回の目的

今回は、会話・施設・NPC表示の流れから一段下へ潜り、**VRAM / CGRAM / OAM / PPU register / DMA / BG tilemap** を採取して、町・フィールド・戦闘画面のマップチップとグラフィック復元へ進めるための足場を作った。

前回までの主線では、`$0A61` active linked list、`C0:AF33` object生成、`C0:B03D -> C0:B100 -> C0:B0C7` OAM構築がほぼ見えている。今回はその延長として、VRAM転送キューとBG tilemapを同時に採取する。

## 1. ROM確認

| size | `2097152` bytes / `0x200000` |

作業上は、既存逆アセンブルと同じく **`C0:0000 == raw 0x000000` のlinear HiROM-like配置** として扱う。Lua v2 の DMA source 推定でも `C0-DF` / `80-9F` mirror をこの前提で `rom_offset_guess` に落としている。

## 2. GitHub current_master からの引継ぎ要点

2026-04-26 final の current master では、13ゴール平均が約87.5%まで進行。特に今回のグラフィック復元に関係するのは以下。

- Goal 13: `$0A61` active linked list、`$0A1F` previous、`$0AA3` sort/depth key、`C0:AF33/AFEC/B03D/B100/B0C7` が接続済み。
- Goal 4: `81:8D87` は `$1569[0..9]` を走査する logical object count aggregator として整理済み。
- Goal 10: 文字表示は実用段階だが、`$718B/$7197` staging raw と最終glyph変換の一部が残る。
- 次に攻めるべき箇所として `C0:B100` と `B294 table` の完全接続、`$0B27` bit定義、`$0A1C` OAM budget単位確定が挙げられている。

今回のv2 Luaでは、Goal 13の object list と VRAM/BG tilemap を同じsnapshotで採取するようにした。これで「画面にいるobject」と「実際に表示されるtile/mapchip」を同じframe単位で突き合わせられる。

## 3. 既存 v1 Lua の重要な修正点

添付 `shinmomo_graphics_mapchip_probe_v1_bizhawk_snes9x_20260501.lua` は方向性として正しいが、Lua上のスコープ問題がある。

- `record_ppu_write()` が `ppu_reg_name()` を参照している。
- しかし、元ファイルでは `function ppu_reg_name(addr)` が後段で定義されており、環境によっては global/local の扱いで `nil` 参照になる可能性がある。
- v2では `local ppu_reg_name` を前方宣言し、`ppu_reg_name = function(addr)` に直している。

## 5. v2で増えた解析価値

### 5.1 DMA source の ROM offset 推定

`dma_trigger_log.csv` に以下を追加した。

- `source`
- `rom_offset_guess`
- `rom_map_guess`
- `vram_word_addr`
- `vram_byte_addr`
- `scene_label`

これにより、町・フィールド・戦闘で発生した DMA が、どのROM資材またはWRAM stagingから来ているかを追える。

重要: `source=7E:xxxx` はWRAM staging転送なので、ROM資材はさらに上流の展開・解凍処理を追う必要がある。`source=Cx:xxxx` / `8x:xxxx` が出れば、ROM直DMAまたはROM側asset pointerの強い手がかりになる。

### 5.2 BG tile usage

各snapshotで `bgN_tile_usage.csv` を出す。これは `bgN_tilemap.csv` を集計し、tile番号ごとの使用数とpalette/priority/flip variantをまとめるもの。

町・フィールド・戦闘の3場面で比較すると、

- 共通UI tile
- フィールド専用地形tile
- 町マップチップtile
- 戦闘背景tile
- window/text layer tile

を分離しやすくなる。

### 5.3 visible object list

各snapshotで `$0A61` active linked list を `visible_objects.csv` として出す。

採取対象:

```text
$0A61 next
$0A1F previous
$0AA3 sort/depth key
$0B25 object type/flags候補
$0AE5 attr候補
$0E27 state候補
$0DA5/$0DE5 X座標系候補
$0D25/$0D65 Y座標系候補
$0BA5/$0C65 周辺raw
```

これで、画面上のNPC・同行者・effect object と OAM/tile側を照合できる。

## 6. 静的 PPU/DMA scan

`data/graphics/20260501_static_dma_ppu_scan.csv` を作成した。

| pattern | count |
|---|---:|
| STA_2115_VMAIN | 5 |
| STA_2116_VMADD | 9 |
| STA_2117_VMADDH | 4 |
| STA_2118_VMDATAL | 5 |
| STA_2119_VMDATAH | 4 |
| STA_2121_CGADD | 0 |
| STA_2122_CGDATA | 0 |
| STA_420B_MDMAEN | 8 |
| STA_420C_HDMAEN | 2 |
| DMA0_BBAD_write | 6 |
| DMA0_A1T_write | 5 |
| DMA0_A1B_write | 4 |
| DMA0_DAS_write | 5 |

注目点:

- `C0:A216` 周辺は、`7E:8000` 転送キューを読み、`$2115/$2116/$4300..$4306/$420B` を設定してVRAMへ流す本線。
- `C0:A151/A170/A185` は、`7E:8000` キューへ転送レコードを積む側。
- `C0:A474` は転送本体ではなく `$036F=1` の更新要求フラグ setter。
- `C0:B0C7` は OAM DMA 転送で、`$0EE9` mirror から `$2104` へ `0x220` bytes を送る。
- `C0:B3DC..B413` は CGRAM DMA 転送で、`$7E:21C2` から `$2122` へ `0x0200` bytes を送る。

## 7. 実行手順

### 7.1 まず場面ごとに分けて採取

可能なら、以下の3回に分けて実行するのが一番きれい。

1. 町画面で `shinmomo_graphics_mapchip_probe_v2_town_20260501.lua`
2. フィールド画面で `shinmomo_graphics_mapchip_probe_v2_field_20260501.lua`
3. 戦闘画面で `shinmomo_graphics_mapchip_probe_v2_battle_20260501.lua`

同一Luaで移動しながら採取してもよいが、その場合 `scene_label=manual_unknown` になる。フレーム番号とBG layout差分で後から分離できる。

## 8. 次に攻める順番

1. 町・フィールド・戦闘それぞれの `frame_summary.csv` を比較し、BG mode / map_base / char_base / bpp を固定する。
2. `dma_trigger_log.csv` から `VRAM_2118/2119` の `source`, `size`, `vram_word_addr` を抜き、VRAM上のどの領域がどの資材で更新されたかを見る。
3. `bg*_tile_usage.csv` で、場面別に使用tile番号を抽出する。
4. `visible_objects.csv` と `oam_sprites.csv` を照合し、NPC/object tileのsourceとmapchip tileを切り分ける。
5. `C0:A151/A170/A185` callerを静的に掘り、`7E:8000` キューへ「どのROM資材を展開して積んだか」を逆追跡する。
6. 次回は `C0:B100` と `B294 table` の完全接続、または `C0:A151` caller全件整理のどちらかが伸びる。

## 9. 今回時点の見立て

グラフィック復元は、いきなりROM内の圧縮資材を静的に当てにいくより、まず runtime で

```text
ROM/WRAM source -> DMA -> VRAM領域 -> BG tilemap -> tile usage -> 画面
```

の骨格を作るのが安全。

今回のv2はこの骨格を採るための虫取り網。町・フィールド・戦闘の `vram.bin/cgram.bin/oam.bin` が揃えば、次の1時間回では、BGごとの `map_base/char_base/bpp` と DMA source cluster を具体的に比較して、マップチップ候補を表に落とせる。
