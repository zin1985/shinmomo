この文書は、`shinmomo_handover_integrated_20260316.md` を土台に、2026-03-18 までの追加解析結果を上書き統合した再開用ハンドオフです。

## 0. 今回の要点
- **`0x41A10 -> 39850/39993`** は target-side の施設/イベント blob 系。
- **`1D8A -> BDE7 -> C379/C931`** は bank82 内の local rule / membership engine 系。
- **`19D5` は施設種別そのものではなく choice layer** とみるのが自然。
- **ledger (`1B15/1B25/1B35`) -> bucket (`BCA5/BCE1/BD06`) -> choice (`1D77/1D87`)** の中間層がかなり見えた。
- **会話本文そのものの一括抽出は未完成**。本文を抜くには `0x41A10` reader hunt と `0x30000` 台 script 帯の解読が最優先。


## 1. 2026-03-18 時点の追加進展

### 1-1. 店/施設UI・ledger 系
- `83:D906` は **purchase commit** として読める。`money -= amount` の直後に `$1621/$1622` を add する。
- `83:D8C1` は **reverse transaction** として読める。`$1621/$1622 -= amount` の後に所持金 add。
- `$1936` は **所持金本体級**、`$1969/$196A` は **今回の取引額ワーク**、`$1E/$1F` は **money helper に渡す amount** とみるのが自然。
- `82:B0B9` は `$1621/$1622` を `1B15/1B25/1B35[$1986]` に flush する **ledger commit helper**。direct caller は未発見で、indirect dispatcher 管理下の可能性が高い。
- `82:96A0` は `1B15/1B25/1B35[$198D] -> 1FC8/1FC9/1FCA` の **ledger load** 本体。
- `82:BCA5 / BCE1 / BD06` は raw money の閾値比較器ではなく、**内部 bucket / class selector** と見る方が自然。

### 1-2. choice / row / descriptor 系
- `1D77` は **choice_id list**、`198B` は **current choice_id**、`19D5` は **主選択肢コード**、`19E5` は **子要素/非トップレベル補助** に見える。
- `19D5` は `0314` と `0315` に落ちる。`0314` は direct row-code、`0315` は bank84/bank82 側 script が読む semantic code 寄り。
- `86:90DB/90E3 -> C0:07FB` は **`[kind, long handler]` の dispatch table**。`19D5[current]` を kind ごとの predicate / runner へ振り分ける層。
- `86:918F / 9196 / 919F` は重い本体ではなく、`1A25`、`1984`、`1B85->0819` を見る **軽量 predicate 群**。
- `86:9145 / 9149 / 914D` は mode `0C / 06 / 0D` 指定 wrapper。`82:CA2E -> C931 -> C54C` の **membership / category 所属判定 runner** を起動する。

### 1-3. bank82 local rule engine 系
- `82:C4F4` は `E0 = 82:BDE7 + (1D8A * 0x0F)` を作る。`[E0]` は **15バイト固定長 local rule-record**。
- `82:C379` は runtime membership table builder。`1D91/1DD1/1E91/1ED1/1F11` を current context と `[E0]+0x0C` から組み立てる。
- `82:C54C` は matcher。`1D89 == 1D91[X]` かつ `1D8A == 1DD1[X]` の行が見つかったとき carry set、`1ED1[X]` を返す。
- `82:C36A` は table 初期化、`82:C564/C578` は挿入シフト、`82:CAA0` は shadow table から active table への bulk loader。
- `1E11 / 1E51 / 1ED1 / 1F11` は、row payload/result/control 列として読むのが自然。特に `1ED1` は result、`1F11` は control byte、`1E11/1E51` は row-linked payload/result low/high 候補。
- `1D8A` は **施設IDそのものではなく local mode/category/subphase**。`C955/C999` state entry 群、`C931` 直判定群、`C50E` account lookup 群、`CCF2/CD38` class modifier 群などに分かれる。
- `CCDE / CCE8` は active slot (`1986`) を `1E11[current_row]` へ落とす軽い stub。
- `CCF2 / CD38` は `1B65[current]` に対する **bitmask set / clear handler**。

### 1-4. `41A10` target-side blob 系
- `0x41A10` first table record 68 は `key=0x74, c1..c5=01,03,02,05,04, target=0x39850`。
- second table record 5 は `key=0x00, c1..c5=05,03,02,02,02, target=0x39993`。
- `0x39847` 開始の A系 9バイト列、`0x39916` 開始の B系 9バイト列が見える。`0x39850` は A系2件目先頭、`0x39993` は B系内部ラベル寄り。
- `0x39839` 近辺には `0x7530` らしき値があり、といちや 30000両条件との関連が濃いが、**コード即値比較より blob/data 側の閾値** とみる方が自然。
- ただし `398xx/399xx` は `C379` builder 側とは別 subsystem。今のところ direct な交点は未発見。


## 2. 2026-03-18 までで認識が変わった点

### 2-1. 追加で進んだ点
- 店/施設の transaction core と ledger subsystem がかなり読めた。
- `19D5` は施設種別そのものではなく、施設内UIの **choice layer** とみる方が自然になった。
- `198D` は固定施設IDではなく、ledger/choice/descriptor をつなぐ **current candidate ID** とみるのが自然になった。
- `82:96A0 -> BCA5/BCE1/BD06 -> 1D77/1D87` という **ledger -> bucket -> choice** の流れが見えた。
- `82:BDE7` local rule-record 群と `39850/39993` target-side blob 群が別 subsystem だと切り分けられた。

### 2-2. 修正が入った点
- **`C955/C999` は DMA/VRAM entry ではない。** 実際は bank82 の local state/script entry 側とみる方が自然。
- **`82:A70B/A70E` の `$1E11,X` は ROM 表ではなく WRAM 表。** 以前の「ROM表を読んでいる」見立ては撤回。
- **`398xx/399xx` blob と `C379 builder` の `[E0]` は同じ record 群ではない。** `[E0]` は `82:BDE7 + 15*1D8A` の local rule-record。
- **`86:918F/9196/919F` は重い本体ではない。** 実際は local flag / global mode / class table を見る predicate 群。
