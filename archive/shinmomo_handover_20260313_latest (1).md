## 目的
重い追跡をいったん区切り、次スレでそのまま続行できるように、現時点の確定事項・有力仮説・未解決点・次の一手を整理する。

---

## 1. 今回まででかなり固いこと

### ROM比較
- 比較対象:
  - `Shin Momotarou Densetsu (Beta).smc`
  - `Shin Momotarou Densetsu (J).smc`
- サイズは両方とも `2,097,152 bytes`
- 差分は約 `1040 bytes`、20クラスタ程度
- 重点候補の比較結果:
  - `0x31C68` 一致
  - `0x31DF8` 一致
  - `0x320C8` 一致
  - `0x442BF` 一致
  - `0x449D0` 一致
  - **`0x41AC0` 周辺に規則的差分あり**

### 重点アドレス
- `0x442BF` = 道具表候補（11バイト固定長仮説）
- `0x449D0` = 装備表候補（16バイト固定長仮説）
- `0x41AC0` = 分岐ディスパッチ表候補（後述。実際は中腹）
- `0x31C68 / 0x31DF8 / 0x320C8` = 店/施設スクリプト帯候補

### `0x41A10..` の正体
- 以前 `0x41AC0` と呼んでいたものは、実際には **より大きい 8バイト固定長表の途中**
- 少なくとも **`0x41A10` から同型8バイトレコードが連続**
- 1レコード形式は概ね:
  - `[key][c1][c2][c3][c4][c5][target_lo][target_hi]`
- `target` は **`0x30000 + target`** とみると自然につながる
  - 例: `1DDF -> 0x31DDF`
  - 例: `20A4 -> 0x320A4`
- これにより:
  - **`0x41A10..` は施設/店/イベント/会話の分岐表本命**
  - `0x31DDF`, `0x320A4` などのスクリプト帯へ飛ぶ

### `0x41A10..` とスクリプト接続
- **`0x31DDF` に接続するレコードあり**
- **`0x320A4` に接続するレコードあり**
  - `0x320A4` 先を読むと `0x320C8` を内包する流れがある
- `0x31DF8` へは、現状は **`0x31DDF` に入った後の同ストリーム途中** と見るのが自然
- `0x31C68` への直結は未発見

### β版→通常版差分の意味
- `0x41AC0` 周辺の先頭数件で、主に **`c1/c2` が `FF FF -> 実値`** に変化
- `target` は変化しない
- したがって `FF FF` は **未設定 target ではなく、条件欄の未確定値** の可能性が高い
- これは
  - 店種別
  - 地域分類
  - 段階値
  - 状態カテゴリ
  のような **小さな分類番号ペア** に見える

## 2. WRAM変数まわりのかなり有力な理解

### `$1F9D / $1F9E`
- **店/施設フェーズ値系** が最有力
- `0x3E480` 近傍:
  - `$1F9D -> $1F9E` に退避
- `0x4EFF0` 近傍:
  - `$1F9D == 0` 判定あり
  - 必要時に `$1F9E -> $1F9D` で復帰
- したがって:
  - `$1F9D` = 作業中の段階/フェーズ値
  - `$1F9E` = 退避/復帰用の元値

### `$1F9D` の変換表
- `0x69FA7` 系で `$1F9D` を変換している
- 変換表:
  - `0x69FB6 = FF 03 00 00 01 02 01 02`
- つまり `$1F9D` は生値のままではなく、**グループ化された段階値** に圧縮される
- `0x41A10..` の `c1/c2` は、この変換後値を見ている可能性が高い

### `$1FAA`
- **サブ機能セット番号** が最有力
- `0x6A13E` 付近で:
  - `$1FAA` を読み
  - `×8`
  - 固定ベースに加算
  - 8バイト表を引く
- 展開先は `$1FAC..` 系
- よって:
  - `$1FAA` 自体、または `$1FAC..` に展開された内容が `0x41A10..` の `c3..c5` 候補

### 暫定対応づけ
- `key` = 施設ID / 会話起点ID / 店種別ID のいずれか
- `c1,c2` = `$1F9D` 系（変換後グループ値含む）
- `c3..c5` = `$1FAA` 由来のサブ機能/施設内モード
- `target` = `0x30000` 台スクリプト先頭

※ ただし **`0x41A10..` を実際に走査比較するコード本体は未発見**。

## 3. `0x41A10..` を読むコードについて

### 未発見のもの
- `0x41A10` / `0x41AC0` を bank88側コードが **直接 indexed 参照する痕跡** は未発見
  - 例: `LDA/CMP $9A10,X`, `long,X` など見つからず
- よって、これは **bank88にある生データ表** であり、
  - bank88自身の直読コードではなく
  - **別bankの汎用ルーチン/インタプリタ** が食べる可能性が高い

### 見つかった周辺情報
- `0x300D3` に **`88:9A05` を指す 24bitポインタ** がある
- これは単独ではなく、**`0x30048` から続く35件の8バイト束** の一要素
- さらに `0x30168` にも **同型の小束** がある
- ただし、同型束の中で **bank88 を指すのは `88:9A05` の1件だけ**
- したがって `88:9A05` は **特殊な資材参照** の可能性が高い

### 汎用読取ルーチン
- raw `0x2FD68..0x2FDB2` に、**24bitポインタ + 記述テーブル方式の汎用読取ルーチン** がある
- これは
  - 3バイト記述子配列を読み
  - field種別ごとに 1/2/特殊ビット抽出を切り替え
  - `$1FC8..` に結果を返す
- ただし、これは **`0x41A10..` の8バイト条件表マッチャー本体ではなさそう**
- それでも「ROM側は汎用テーブル読取器を多用している」という設計思想が見えた

## 4. `0x4F` 命令と selector の整理

### 命令番号
- bank89 のディスパッチ表から:
  - **opcode `0x4F` -> handler `89:9A44`**

### `0x4F` の役割
- `0x4F` は会話分岐命令というより、**UI/施設画面で RAM上の現在レコードから field を抜いて表示資材にする命令**

### 実スクリプト例からの理解
実際の使用例では:
- `4F 01 23 19 xx` → selector 1, base `$1923`
- `4F 02 23 19 xx` → selector 2, base `$1923`
- `4F 03 3B 19 xx` → selector 3, base `$193B`
- selector 0 の例も別系列で確認済み

### selector テーブル
- selector 0 → `C4:06D6`
- selector 1 → `C4:06BE`
- selector 2 → `C4:0000`
- selector 3 → `C4:7BF9`

### selector の暫定ラベル
- **selector 1 = 商品系**
- **selector 2 = 装備系 / 特殊商品系**
- **selector 3 = キャラ系**
- selector 0 は一覧/基本表示スキーマ寄りだが未確定

---

## 5. `$1923` と `$193B` の役割

### `$1923`
- **現在選択中の物品レコードID** が最有力
- `selector 1/2` の入力元
- 店の商品、装備候補、特殊物品候補などの共通入口っぽい

### `$193B`
- **現在対象のキャラ/主体ID** が最有力
- `selector 3` の入力元
- 候補ID表から順に入れ替えて存在/参加可否チェックしている痕跡あり

### 実用上の結論
- `$1923` → 物品側
- `$193B` → キャラ側
- これで selector 1/2/3 を
  - 商品系
  - 装備/特殊商品系
  - キャラ系
  と見る筋がかなり強い

---

## 6. `80DEFB` と `81DAC9`

### `80DEFB`
- **商品系スキーマ `C4:06BE` 用の field抽出ラッパ**
- 戻り値は文字列ではなく、**商品/物品属性の数値**
- 例: 種別、装備可否、表示カテゴリ等の可能性

### `81DAC9`
- **キャラID (`$193B`) から 1バイトの表示コードを作るルーチン**
- 文字列ポインタではない
- 顔/記号/アイコンなどの **キャラを表す表示コード** の可能性が高い

---

## 7. `7E:718B` 周辺の意味

### 今分かっていること
- `89:841E / 89:8424`:
  - 1バイト表示コードを `7E:718B..` に詰める
- `89:8443 / 89:845E`:
  - `7E:7189..718B` から表示コードを読み出す helper
- この近傍には、**`7E:718B` の値を別表でタイル番号へ再変換する処理は見えていない**

### 現時点の理解
- `7E:718B..` は **UI表示コードの staging buffer**
- 値は「変換待ちの抽象ID」より、**かなり表示寄りのコード**
- `81DAC9` の戻り値は、このバッファにそのまま積まれることから、
  - **キャラ表示記号/アイコン/顔種別の内部コード** の可能性が高い

### ただし未確定
- `7E:718B` の値が
  - 文字コードなのか
  - 既にタイル番号相当なのか
 までは未確定

## 8. `89:845E` の行き先

### 直接caller
- `89:823B`
- `89:826D`
- `89:8291`

### 挙動
- `89:826D`:
  - `JSR $845E`
  - `STA ($9C)`
- `89:8291`:
  - `JSR $845E`
  - `STA ($9C)` と `STA ($9C),Y`
- `89:823B`:
  - `845E` の結果を `A2/A3` 経由で再び `7E:718B` 側へ戻すルート

### 重要点
- `($9C)` は固定アドレスではなく、**スクリプト命令が引数で渡す RAM書込先ポインタ**
- したがって `845E` の値は、まず **スクリプト指定の作業先RAM** に書かれる
- **tilemap バッファ / DMA 元へ直接行くわけではない**

## 9. いまの全体像

### 高確度
- `0x41A10..` = 8バイト固定長の条件分岐表本命
- `target` は bank83 のスクリプト帯へ飛ぶ (`0x31DDF`, `0x320A4` 等)
- `c1/c2` は `$1F9D` 系段階値が濃い
- `c3..c5` は `$1FAA` 系サブ機能が濃い
- `0x4F` = UI/施設画面用の field抽出命令
- selector 1/2/3 = 商品系 / 装備系 / キャラ系
- `$1923` = 物品側レコードID
- `$193B` = キャラ側ID

### まだ未発見
- **`0x41A10..` を実際に走査して条件比較し、末尾2バイト target を抜く本体コード**
- `7E:718B` / `($9C)` の先にある最終 tilemap / DMA 流れ
- `0x31C68` への直結ルート

## 10. 次スレでの優先タスク

### 最優先
1. **`0x41A10..` 条件表を読む本体コードの発見**
   - 条件:
     - 8バイト刻み
     - `key/c1..c5` 比較
     - 末尾2バイト target 抽出
   - 方向:
     - bank88直読ではなく、**別bankの汎用ループ/インタプリタ** を疑う
     - `0x30048..0x30168` の 8バイト束を読む側も候補

2. **`($9C)` 宛先実例の列挙**
   - `89:8108` 系ディスパッチャを使う実スクリプト例を拾う
   - `($9C)` にどのRAMが入るかを一覧化
   - これで `845E` の出力先が
     - 文字列作業バッファ
     - 一覧行バッファ
     - tilemap staging
     のどれかを切り分ける

3. **`7E:718B` / `($9C)` の先の最終描画経路**
   - `89:8443 / 89:845E` caller の上流・下流をさらに追う
   - tilemap/DMA元に乗る箇所を特定

### 次点
4. **selector 0 の実画面ラベル確定**
   - selector 0 を含む `0x4F` 実スクリプト列の文脈を掘る

5. **`0x31C68` 接続の追跡**
   - `0x31DDF / 0x320A4` 以外の `target` レコードとの接続を広げる

## 11. 作業時の注意
- `0x41AC0` だけを切り出して扱うと誤認しやすい。**表本体は `0x41A10` から見ること**
- `0x4F` は会話分岐というより **UI/施設の field抽出命令** と理解したほうが混乱が少ない
- `80DEFB` や `81DAC9` は **文字列そのものを返していない**
- `7E:718B` は現時点では **表示コード staging buffer** として扱うのが安全
