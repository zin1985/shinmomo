## 0. 今回の更新結論

GitHub上の最新版 `handover/current_master.md` は、2026-03-25時点の **統合監査版** として採用する。内容上は、3/24版を母体にしつつ、3/16・3/18で確立していた表示VM、DMA、item/equip true base、店/施設UI、ledger、local rule engine を復帰させたもの。

今回の追加解析では、13番目の目標である **NPC大量表示時の処理軽減ロジック** に対して、C0/B0xx〜B9xx周辺から以下を新規に強めた。

- `C0:B0C7..B0FF` は **OAM DMA本体** とみてよい。
- `C0:B100..B25C` は **スプライト/OAMエントリ構築 + 画面外クリッピング + OAM空き枠チェック** の中核候補。
- `C0:B30F..B4DD` は **VRAM/CGRAM/タイルキャッシュ系のスロット管理** 候補。
- `C0:B52C..B7A6` は **最大32系統の動的キャッシュ/参照カウント/スロット確保** 候補。
- `C0:B7A7..B8F1` は **キャラ/パターン定義を読み、必要なら転送キューまたは即時VRAM書込へ流す資材転送系** 候補。

これにより、13番「NPC大量表示時の処理軽減ロジック」は、従来の **28%** から **42%** に更新する。まだ「NPC走査そのもの」までは未到達だが、**表示負荷を減らす後段の仕組み** はかなり具体化した。

## 2. 現在の全体像

### 2-1. 三本柱

#### A. 店 / 施設 / 分岐 / UI 系

- `0x41A10..` は8byte固定長条件表本命。
- 概形: `[key][c1][c2][c3][c4][c5][target_lo][target_hi]`
- `target = 0x30000 + target` と読むのが自然。
- `0x41A10 -> 0x39850 / 0x39993` は target-side blob 系。
- `0x41D20` は first/second table 間の remap / skip metadata 候補。
- `0x41D38` は second-table header らしき独立8byte。
- `0x39847 / 0x39916` は A系 / B系の9byte小レコード列。
- `1D8A -> BDE7 -> C379/C931` は bank82 local rule / membership engine。
- `19D5` は施設種別そのものではなく choice layer。
- `ledger (1B15/1B25/1B35) -> bucket (BCA5/BCE1/BD06) -> choice (1D77/1D87)` の流れを採用。

#### B. 文字 / 表示 / descriptor / queue 系

- 主線は bank89。
- `89:9D4D -> 9D91 -> 9E10 -> 9DE5 -> 9EC7 -> A5AF/A5C8`
- `09 token` は AE3A直通 + family root fallback の二段ディスパッチ。
- `82:FD67` は汎用 field 抽出器。
- `82:FE0E` は selector0 special path。
- `81:DAC9` は固定表直引きではなく char schema + 状態依存補正。
- `7E:718B` は描画終点ではなく operand stack / staging。

#### C. 会話本文 / token / scene 系

- `0x70000` 台は descriptor / pointer / page header / small dictionary 側。
- `0x30000` 台は script / event 本体寄り。
- 会話本文の一括抽出は未完成。
- ED直前帯は `0x73D3B..0x7575B`。
- entry 15 = `0x73D3B..0x746EE`
- entry 16 = `0x746EF..0x751DE`
- entry 17 = `0x751DF..0x7575B`
- 避難民11本 macro 辞書列は、3/24版の更新見解を採用。
- `0x7D100..0x7D1DC` は旧「お浜・銀次5発話」固定説を破棄し、複数scene連結帯とみる。

## 4. 今回のNPC大量表示軽減 追加解析

### 4-1. 新しく強くなった仮説

NPC大量表示時の軽減は、おそらく単一の「NPC数制限ルーチン」だけではなく、次の複数層で成立している。

1. **NPC/オブジェクト候補の走査層**
   - まだ本体未発見。
   - ここが「どのNPCを処理対象にするか」を決める上流。
2. **画面内判定 / スプライト片ごとのクリッピング層**
   - `C0:B100..B25C` が本命候補。
   - 画面外のスプライト片はOAMに書かずスキップする。
3. **OAM空き枠 / スプライト枚数管理層**
   - `$0A1C`, `$1109`, `$110B`, `$110C` 周辺。
   - 残りOAM枚数や高位OAM属性のパックを管理している。
4. **動的VRAM / パターンキャッシュ層**
   - `C0:B30F..B4DD`, `C0:B52C..B7A6` が候補。
   - 同じキャラ/パターンを再転送しないためのスロット・参照カウント管理らしい。
5. **実転送層**
   - OAM DMA: `C0:B0C7..B0FF`
   - VRAM DMA queue: `80:A151/A170/A185 -> 7E:8000 -> 80:A1B9/A216相当`
   - CGRAM DMA: `C0:B3DC..B426`

## 5. OAM DMA本体

### 5-1. `C0:B0C7..B0FF`

この範囲は **OAM DMA本体** とみてよい。

根拠:

- `$0A1B` を dirty flag として見る。
- `$0A1B != 0` のときだけ DMA を実行。
- DMA source は `$000E:E9`。
- DMA destination は PPU `$2104`。
- 転送サイズは `$0220` byte。
- `$2102/$2103` を0にしてOAM先頭から転送。
- 転送後 `$0A1B` をクリア。

構造:

```text
C0:B0C7  LDA $0A1B
C0:B0CA  BEQ $B0FF
C0:B0D5  A9 E9      source low = E9
C0:B0DA  A9 0E      source high = 0E
C0:B0E4  A9 20      length low = 20
C0:B0EC  A9 02      length high = 02
C0:B0F1  STZ $2102  OAMADDL
C0:B0F4  STZ $2103  OAMADDH
C0:B0FA  STA $420B  DMA start
```

解釈:

- `$0EE9..` はOAM lower table相当。
- `$10E9..` はOAM high table相当。
- `$0A1B` は「OAM更新あり」フラグ。
- `$0A1C` はOAMエントリ残数または使用可能枠数に近い。

## 6. スプライト片構築 + 画面外クリッピング

### 6-1. `C0:B100..B25C`

この範囲は **スプライト定義を読み、画面内にある片だけOAMへ積む処理** とみるのが自然。

主な流れ:

1. `X` = オブジェクト/NPC index。
2. `$0AE5,X` から sprite pattern/animation index を取得。
3. `B294` のポインタ表から定義データを引く。
4. 先頭byteを `$24` に入れる。`$24` はスプライト片数候補。
5. `$0A1C` と `$24` を比較。`$0A1C < $24` なら return。残りOAM枠が足りなければ描画しない可能性。
6. 各スプライト片についてX/Y座標を計算。
7. 画面外ならOAM書込をスキップ。
8. 画面内なら `$0EE9..` に4byte OAM entryを書き、`$10E9..` に高位bit/size/属性を詰める。

### 6-2. X方向クリッピング

```text
C0:B171  ADC $0BA5,X  ; object X low 加算
C0:B179  ADC $0BE5,X  ; object X high 加算
C0:B17C  AND #$0F
C0:B180  BEQ in_range
C0:B182  CMP #$0F
C0:B184  BNE skip
C0:B186  LDA $06
C0:B188  CMP #$C0
C0:B18A  BCS in_range
C0:B18C  JMP $B24C ; skip piece
```

- `$0BA5/$0BE5` は対象のX座標系。
- high nibbleが `0` なら通常画面内。
- high nibbleが `0xF` かつ lowが `>= C0` なら左端付近の負座標として許容。
- それ以外は画面外としてスキップ。

### 6-3. Y方向クリッピング

```text
C0:B19C  SBC $0CE5,X
C0:B1A1  ADC $0C65,X
C0:B1A8  ADC $0CA5,X
C0:B1AD  BEQ in_range
C0:B1AF  CMP #$0F
C0:B1B1  BNE skip
C0:B1B3  LDA $09
C0:B1B5  CMP #$F0
C0:B1B7  BCS ok_negative_edge
C0:B1B9  JMP $B24D ; skip piece
C0:B1BC  LDA $09
C0:B1BE  CMP #$E0
C0:B1C0  BCS skip
```

- `$0C65/$0CA5/$0CE5` は対象のY座標/補正/基準値候補。
- Yが `0xE0` 以上の場合は基本的に画面外扱い。
- `0xF0` 以上は負座標側の端として一部許容している。
- SNES OAMのY=E0非表示慣習とも整合する。

### 6-4. OAM entry 書込

```text
C0:B225  LDX $1109
C0:B228  LDA $06       ; X
C0:B22A  STA $0EE9,X
C0:B22F  LDA $09       ; Y-1
C0:B231  STA $0EE9,X
C0:B235  LDA $0F       ; tile
C0:B237  STA $0EE9,X
C0:B23B  LDA $10       ; attr
C0:B23D  STA $0EE9,X
C0:B241  STX $1109
C0:B247  DEC $0A1C
```

- `$1109` はOAM lower table書込ポインタ。
- 1スプライト片ごとに4byte進む。
- `$0A1C` は残りOAM sprite枠カウンタに近い。
- `$110B/$110C` は high OAM table のbyte位置/bit位置を管理している。

## 7. OAM初期化 / 非表示埋め

### 7-1. `C0:B08D..B0C6`

ここはOAM buffer初期化とdirty flag setに見える。

```text
C0:B096  STZ $0EE9,X
C0:B099  STA $0EEA,X ; E0
C0:B09C  STZ $0EEB,X
C0:B09F  STZ $0EEC,X
C0:B0C0  A9 01
C0:B0C2  STA $0A1B
```

- Y座標に `E0` を入れてスプライトを非表示化している可能性が高い。
- `$1E` 個分を初期化している。
- 最後に `$0A1B = 1` でOAM DMA対象にする。

## 8. 動的VRAM / タイルキャッシュ候補

### 8-1. `C0:B30F..B3D6`

`C0:B30F` は `7E:21C2` を0x200 byteクリアし、さらに `7E:23C2 / 23CA / 23D2` と `$1112` を8件分クリアしている。

- `7E:21C2` はCGRAM bufferまたはパレット/色テーブル staging 候補。
- `7E:23C2` = asset id / pattern id
- `7E:23CA` = bank/scene id または `$035F` 由来のグループID
- `7E:23D2` = slot length / tile count
- `$1112` = 8slot分の参照カウント

### 8-2. `C0:B3DC..B426` CGRAM DMA候補

```text
C0:B3DC  LDA $111A
C0:B3DF  BEQ end
C0:B3E1  LDA $0AE6
C0:B3E4  BEQ end
source = 7E:21C2
dest   = $2122
size   = 0x0200
C0:B40E STZ $2121 ; CGADD
C0:B413 STA $420B ; DMA start
C0:B416 STZ $111A
```

- `$111A` はCGRAM更新dirty flag。
- `$0AE6` はCGRAM更新許可/画面状態フラグ。
- `7E:21C2` から PPU `$2122` へ0x200 byte DMA。
- これはパレット更新系の本体としてかなり強い。

### 8-3. `C0:B427..B4DD` 8slotキャッシュ管理

`C0:B427` は、指定された asset id と現在の `$035F` をキーに、8件のスロットを探す。

- 既存スロットがあれば `$1112,X` をインクリメントして再利用。
- 空き連続スロットがあれば確保。
- `B35C` を呼んで `7E:21C2` 側に関連データをコピー。
- 見つからなければ carry set / `FF` で失敗。

**同じNPC/キャラ資材を何度もVRAM/CGRAM転送しないための軽量キャッシュ** とみるのが自然。

### 8-4. `C0:B4DE..B515` 参照カウント解放

- Aがslot index。
- `$1112,X` をデクリメント。
- 0になったら `7E:23C2 / 23CA / 23D2 / $1112` をクリア。

## 9. 最大32系統のキャッシュ/パターン資材管理

### 9-1. `C0:B52C..B606`

`C0:B52C` は、以下の32件テーブルをクリアする。

- `7E:24FA`
- `7E:251A`
- `7E:253A`
- `7E:255A`
- `7E:257A`
- `7E:23DA..24D9` は `FF` 埋め

`C0:B58D..B606` は、asset定義を読み、既存スロットを探し、なければ空きslotを確保して `B893` で転送を積む。

- 32件分の「動的キャラ/パターン資材」キャッシュ管理に近い。
- `7E:257A` は参照カウント。
- `7E:251A/24FA/253A/255A` は asset id / bank / width / height / group などのキー列。
- `7E:23DA` はタイル占有マップまたは動的VRAM tile slot map。

### 9-2. `C0:B654` 既存スロット一致判定

一致条件:

```text
7E:251A,X == $111C
7E:24FA,X == $035F
7E:253A,X == $1122
7E:255A,X == $1121
```

同じ asset id、同じグループ、同じ寸法/派生情報なら再利用。再利用時は `7E:257A,X` をインクリメント。

### 9-3. `C0:B6D5..B729` タイル空き領域探索

`7E:23DA` を走査し、`FF` 連続領域を探している。

- `$1121` から必要tile数を計算。
- `$1125` が探索上限。
- `7E:23DA,X` が負値/`FF` なら空き候補。
- 空きが足りなければ carry set。

動的VRAMタイル領域の空きスロット探索とみる。

## 10. 資材定義読込 + VRAM転送キュー投入

### 10-1. `C0:B7A7..B7F9`

指定された asset id から定義レコードを読み、以下を取り出す。

- `$0F` = 転送先/VRAM address候補
- `$0C` = 転送サイズ/幅高さ候補
- `$06` = source pointer/offset候補
- `$08` = bank/flags候補
- `$09` = 転送モード/形式候補
- `$1123` = 転送形式 dispatch index

`B8F2` により、`$1123` に応じて転送前処理が分岐する。

### 10-2. `C0:B7FA..B8F1`

- `$03AB != 0` の場合は `80:A151/A170/A185` を使って `7E:8000` 転送キューへ積む。
- `$03AB == 0` の場合は `80:A30F` でキューを空にしてから、PPU `$2118/$2119` へ直接書く。

通常時は転送キュー方式、特殊状態では即時VRAM書込方式。NPC大量表示時は、動的タイルの必要分だけキュー化される。

## 11. NPC大量表示軽減に関する現時点のモデル

```text
NPC/オブジェクト走査（未発見）
  ↓
対象NPCのsprite/asset id決定
  ↓
C0:B58D/B5xx: 32slotキャッシュ確認
  ├─ 既存資材あり → refcount++、転送なし
  └─ 未登録 → 7E:23DAから空きtile領域探索
       ├─ 空きあり → VRAM転送キューへ積む
       └─ 空きなし → 表示失敗/抑制の可能性
  ↓
C0:B100: sprite片定義を読み、画面内判定
  ├─ 画面外 → OAM書込をスキップ
  ├─ OAM残枠不足 → 描画せずreturn
  └─ 画面内 → $0EE9/$10E9 OAM bufferへ書込
  ↓
C0:B0C7: dirty flagが立っていればOAM DMA
```

現時点で見えている軽減要素:

1. 画面外スプライト片はOAMに積まない。
2. OAM残枠不足時は描画を諦める。
3. 同一資材はVRAMへ再転送せずキャッシュ再利用する。
4. 参照カウントで使われなくなった資材slotを解放する。
5. VRAM転送は `7E:8000` queue 化され、VBlank予算内で消化される。
6. CGRAM更新もdirty flag方式で必要時だけDMAされる。
