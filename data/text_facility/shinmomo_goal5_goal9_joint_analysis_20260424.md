## 0. 今回の目的

文字コード解析の成果を前提に、13目標のうち次を同時に伸ばす。

- **Goal 5: 店・施設UIのレコード構造を確定する**

---

## 1. 採用する前提

GitHub `handover/current_master.md` の採用方針に従う。

- `82:FD67` は汎用 field 抽出器。
- `89:9A44` / opcode `0x4F` は、RAM上の現在レコードから field を抜いて表示資材にする命令。
- `19D5` は施設種別そのものではなく choice layer。
- `ledger (1B15/1B25/1B35) -> bucket (BCA5/BCE1/BD06) -> choice (1D77/1D87)` の流れを維持。
- `0x70000` 台は descriptor / pointer / page header / small dictionary 側。
- `0x30000` 台は script / event 本体寄り。
- `0x0823CD..0x0845FF` は、文字コード解析により **商店・施設・ミニゲーム・探索反応のシステム文クラスタ** として扱う。

---

## 2. 今回の結論

`0x0823CD..0x0845FF` は、以下のように **施設UI機能の順に近い並び** になっている可能性が高い。

1. 商店の呼び込み・カテゴリ導入
2. 販売制限・薬・茶店券・道具説明
3. 人気度 / 品揃え変動 / 店舗解禁
4. 会計・所持金不足・装備購入応対
5. 装備付け替え / 下取り / 差額払い
6. 預かり・といち屋・金庫・持ち物制限
7. 医者 / 治療料金
8. 宿屋 / 所持金不足 / 宿泊
9. 人気度80・宿屋解禁・探索反応
10. 夜店ミニゲーム説明・景品・進行ガイド

つまりこの帯は、単純な会話文ではなく **施設UIの状態遷移ごとに出る定型文パック** と見るのが自然。

## 5. 今回見えた構造仮説

### 仮説1: 施設文は「機能ごとの表示文パック」として並ぶ

`0x0823CD..0x0845FF` は、物語順ではなく、施設UIの機能順に近い。

大まかな流れ:

```text
店入口
 -> 商品カテゴリ
 -> 販売制限
 -> 会計
 -> 装備/下取り
 -> 預かり/金庫
 -> 医者
 -> 宿屋
 -> 人気度/探索反応
 -> ミニゲーム
 -> 景品/進行ガイド
```

これは、ROM全体のテキスト配置が「シナリオ順」よりも **マップ・場面・機能クラスタ順** であるという既存見解と一致する。

### 仮説2: Goal 5 のコード側 `ledger / choice` と、文字資材側の店文が対応する

特に以下は接続候補として強い。

| コード側 | 文字資材側 | 解釈 |
|---|---|---|
| `83:D906 purchase commit` | `0x0826AC..0x08271F` | 購入成立 / 所持金不足 / 会計 |
| `83:D8C1 reverse transaction` | `0x0828EE..0x0829FD` | 下取り / 差額 / 付け替え |
| `82:B0B9 ledger commit` | `0x082D00..0x0830FF` | 預かり・金庫・引き出し |
| `82:96A0 ledger load` | `0x082F00..0x0830FF` | 金庫内容確認 / 空判定 |
| `BCA5/BCE1/BD06 bucket/class selector` | `0x0825C2..0x082635` / `0x083400..0x0835FF` | 人気度・条件解禁 |

まだ direct call は取れていないが、機能対応としてはかなり自然。

### 仮説3: `02xx` は店文仕様化の最後の壁

店文帯では `02B0`, `02B8`, `02C5`, `02D1`, `02D3`, `02D4`, `02D5`, `02D7`, `02D8`, `02D9` などが頻出する。
これは単なる文字ではなく、以下のいずれかの可能性が高い。

- 商品名 / 装備名の差し込み
- 金額 / 数値の差し込み
- キャラ名 / 主体名の差し込み
- 店舗名 / カテゴリ名の定型句
- 確認文の共通マクロ

したがって次に Goal 5 / 9 を伸ばすなら、`02xx` を辞書参照・動的差し込みとして扱い、静的本文から切り離すのがよい。
