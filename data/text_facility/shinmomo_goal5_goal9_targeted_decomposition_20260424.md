## 1. 0x0830DE..0x08310F 人気度30未満・治療費2倍文

### raw
```text
0830DE: 18 47 18 48 18 49 D2 00 FE 54 51 BC BB F5 9F A8
0830EE: AE EE 91 A1 B2 A5 00 53 DE 91 A5 FF A0 B7 F8 92
0830FE: AA BC 01 A9 FF B6 92 A9 B1 A5 50 A4 F5 DA A2 9B
```

### 文字単位の芯
- `18 47 18 48 18 49` = `人気度`
- `54 51` = `30`
- `BC BB F5 9F A8` = `をわったの`
- `91 A1 B2` = `いつも`
- `53 DE 91 A5` = `2ばいに`
- `A0 B7 F8 92 AA BC` = `ちりょうひを`
- `B6 92 A9 B1 A5 50 A4 F5 DA A2 9B...` = `らうはめに なってし...`

### 推定復元
```text
人気度が 30をわったので
いつもの 2ばいに
ちりょうひを はらうはめに なってしまった！
```

### 未解決 / 注意
- `D2 00 FE` は静的デコードでは `ぐ / END / FE` に見えるが、文脈上は `が` または文表示制御込みの `が` 相当。
- `AE EE` は静的表では `ま + 未確定` になるが、文脈上は `で` または `ので` 周辺の制御・辞書・濁点処理が混じっている可能性。
- `01`, `FF` は本文ではなく制御バイトとして扱うべき。

### 機能分類
- Goal 5: 医者UIの料金補正表示。
- Goal 9: 人気度システム文。人気度 `0..29` で医者代2倍。

## 2. 0x083420..0x083458 人気度80超え・宿屋半額文

### raw
```text
083420: 7A 01 47 18 48 CD 0F 8F 00 8A C8 02 64 18 49 D0
083430: 50 FF 59 51 BC 99 93 9F A8 BE 9E 01 3A 10 C8 1A
083440: F2 1A F3 6F 10 EF FE 18 18 80 19 B0 19 83 F6 A5
083450: 50 A3 AE 24 00
```

### 文字単位の芯
- `02 64` = 辞書参照。文脈上 `人気` 候補。
- `18 49` = `度`
- `D0` = `が`
- `59 51` = `80`
- `BC 99 93 9F A8 BE` = `をこえたので`
- `1A F2 1A F3` = `半分`
- `18 18 80 19 B0 19 83` = `段ー宿屋` と崩れて見えるが、文脈上は `値段で 宿屋` 近辺。
- `F6 A5 50 A3 AE` = `ゃに とま` と崩れるが、文脈上は `屋に とま...`

### 推定復元
```text
人気度が 80をこえたので
半分の値段で
宿屋に とまれるようになった！
```

### 未解決 / 注意
- `02 64` は `人気` の辞書候補としてかなり強い。
- `10 C8`、`6F 10 EF FE`、`24 00` は本文ではなく、改行・助詞・送り・行終端などの制御/辞書混在とみる。
- ROM文面が `80をこえた` なので、条件は表示上 `>80`。攻略情報の `81以上` と整合する。

### 機能分類
- Goal 5: 宿屋UIの人気度割引表示。
- Goal 9: 人気度80超えシステム通知。

## 4. 0x39839 / 0x39850 / 0x39993 30000両条件 blob 再検証

### 0x39839
```text
039837: 3E
039838: 42
039839: 30 75  ; little endian 0x7530 = 30000
03983B: 00
03983C: 70 F0 48 00
039840: 3E 42 30 76 00 22 E8
```

- `0x39839` に `0x7530 = 30000` が存在する。
- 直前直後がコード即値比較というより、target-side blob / 条件データの一部に見える。
- Web攻略情報では、といちやは預け金が30001両以上になると30000両で雇える。
- よって `0x7530` は「雇用費用 30000両」または「預け金しきい値判定の表示・条件値」候補。
- ただし加入条件が攻略上 `30001以上` の場合、ROM内には別途 `30000を超える` として実装されている可能性がある。

### A系 9バイト record: 0x39847 start
基本形:
```text
38 05 01 <id> 32 <lo> <hi> 9A F0
```
例:
```text
0x39847: 38 05 01 01 32 03 01 9A F0
0x39850: 38 05 01 02 32 10 07 9A F0
0x39859: 38 05 01 04 32 1A 0A 9A F0
```

### B系 9バイト record: 0x39916 start
基本形:
```text
38 05 01 <id> 32 <lo> <hi> DB F0
```
例:
```text
0x39916: 38 05 01 01 32 04 01 DB F0
0x3991F: 38 05 01 02 32 11 07 DB F0
0x39928: 38 05 01 04 32 1B 0A DB F0
0x39993: 38 05 01 10 32 8B 27 DB F0
```

### A/B比較
- A系とB系は同じ `<id>` 群を持つ。
- B系の `<lo><hi>` は多くが A系 +1。
- tail が A系 `9A F0`、B系 `DB F0` で異なる。
- `0x39850` は A系 record 2。
- `0x39993` は B系 record 16。

### 現時点の判断
- `0x39839` の `30000` と `0x39847/0x39916` の9バイト列は近接しており、といちや加入条件との関連は強い。
- ただし 9バイト列本体は「30000両条件そのもの」ではなく、条件成立後に呼ばれる target-side event/action table の可能性が高い。
- `0x41A10 -> 0x39850 / 0x39993` は、施設/イベント blob へのラベル参照と見る。

## 5. 0x09BF28..0x09BF5A 怨みの洞窟 名前入力差し込み

### raw
```text
09BF28: 91 EF A0 DE BD 96 72 7B 96 B6 E8 7C 91 A4 A6 01
09BF38: A8 AD D7 21 D4 9F 09 DF 40 67 1A 9A 91 A2 50 FD
09BF48: 53 19 5F 1A F4 A5 AD 13 72 0F 26 2C 41 DB 34 80
09BF58: 19 FF 1A
```

### 静的に読める芯
- `91 ... A0 DE BD 96 ...` = `いちばんき...` 候補
- `7B 96 B6 ... 7C` = `（きらい）` 周辺候補
- `1A9A 91 A2` = `続いて`
- `53` = `2`
- `19 5F` = `番`
- `1A F4` = `目`
- `A5` = `に`

### Web会話集との照合による復元
```text
いちばん きらいな人の 名前を お聞かせください！
続いて 2番目に 嫌いな人の 名前を お聞かせください！
最後に 3番目に 嫌いな人の 名前を お聞かせください！
```

### 静的復元が崩れる理由
- この範囲は通常の本文だけでなく、名前入力UIの呼び出し、順位差し込み、質問文、入力結果の保存が混ざる。
- そのため `0x09BF28..0x09BF5A` だけを文字表で読むと、`R`, `{21}`, `{40}` などが混入する。

### runtime hook方針
既存の `84:A1F3` / `84:B5E3` 表示フックに加えて、以下を追加して取る。

#### 目的
- 表示された実文
- lineID `$12B4`
- 現在文字 `$12B2`
- table `$12A9`
- 名前入力直前/直後の RAM
- 名前として保存された文字コード列

#### 追加候補RAM窓
```lua
RAM_WINDOWS = {
  { name = "dlg",      base = 0x12A0, size = 0x40 },
  { name = "stream",   base = 0x1180, size = 0x40 },
  { name = "namebuf1", base = 0x1200, size = 0x80 }, -- 仮。名前入力時に差分確認
  { name = "work1D",   base = 0x1D00, size = 0x100 },
}
```

#### 手順
1. ほほえみの村/怨みの洞窟入口直前のsavestateを作る。
2. markerを `urami-before-input` にする。
3. 入口NPCに話しかけ、質問文3本を表示。
4. 1人目/2人目/3人目にそれぞれ特徴的な名前を入力する。
   - 例: `ああああ`, `いいいい`, `うううう`
5. `84:B5E3` のflushログで、質問文とlineIDを取る。
6. 入力後の RAM diff で、名前保存先を特定する。
7. 次に敵エンカウント表示まで進め、名前が敵名へ反映される時のlineIDとRAMを取る。
