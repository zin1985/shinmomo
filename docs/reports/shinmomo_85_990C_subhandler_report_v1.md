# 新桃太郎伝説 $85:990C subhandler 解析 v1

## 結論

`$85:990C` は、`opcode $02 / subcode $32` の本体です。
ただし、ここで `02 32` の後続バイトを直接読むのではなく、**scheduler/task callback を登録する入口**でした。

入口処理は以下です。

```asm
85:990C  INC $1268
85:990F  STZ $0365
85:9912  REP #$20
85:9914  LDA #$9924
85:9917  STA $0F
85:9919  SEP #$20
85:991B  LDA #$32
85:991D  STA $0D
85:991F  JSL $80:AC1E
85:9923  RTL
```

つまり、`$0F/$10 = $9924`、`$0D = $32` をセットして、`$80:AC1E` により `$85:9924` をcallbackとして登録しています。

## 重要な発見

前回の `02 32 10 07 9A F0` について、今回かなり認識が変わりました。

`opcode $02` handlerは、`02 32` の **2バイトだけ** を消費し、`$85:990C` へ飛びます。
`$85:990C` も後続の `10 07 9A F0` を直接読んでいません。

そのため、`10 07 9A F0` は `$85:990C` のtail operandではなく、**VM stream上の後続命令/後続データ** と見る方が自然です。

```text
38 05 01        ; opcode38 + operand 0x0105
02 32           ; opcode02 + subcode32 -> $85:990C
10 07           ; 次のVM命令候補
9A F0           ; さらに後続。opcode/data/branch片のどれかは未確定
```

## $85:9924 callback/router

`$85:9924` は、`$1924` と `$1923` を見て処理を切り替えるrouterです。

```asm
85:9924  LDA $1924
85:9927  BNE $994A       ; nonzeroならcleanup
85:9929  LDX #$00
85:992B  LDA $9952,X     ; selector table key
85:9930  CMP $1923
85:993A  LDA $9953,X     ; callback pointer
85:9943  JSL $80:AD1D    ; callback差し替え
```

`$1923` selector table は以下です。

| `$1923` | callback | 静的な見立て |
|---:|---|---|
| `$3B` | `$85:998E` | `$1620 = $17` をセットして共通処理へ |
| `$3C` | `$85:9992` | `$1620 = $19` をセットして共通処理へ |
| `$3D` | `$85:9996` | `$1620 = $18` をセットして共通処理へ |
| `$3A` | `$85:9962` | `$1958/$194B` を使う複合処理。失敗時cleanup |
| `$0D` | `$85:999A` | `$1620 = $192A` をセットして共通処理へ |

## scheduler 周辺の意味

`$80:AC1E` は、現在bankのcallbackをscheduler配列へ登録する処理です。
`$0D` が type/priority/id、`$0F/$10` がcallback address、caller bankがcallback bankとして使われます。

登録先として以下の配列が見えます。

```text
$0499,X : task type / priority らしき値
$0519,X : callback low
$0559,X : callback high
$0599,X : callback bank
$03D9/$0419/$0459/$04D9/$05D9 : task link / countdown / aux 系
```

`$80:AD1D` は現在taskのcallback pointer差し替え、`$80:AD35` は現在task削除/cleanupと見てよさそうです。

## $85:9962 / $85:998E.. の役割

`$85:998E/$9992/$9996/$999A` は、`$1620` に値を入れてから共通処理へ流します。

```text
$3B -> $1620 = $17
$3C -> $1620 = $19
$3D -> $1620 = $18
$0D -> $1620 = $192A
```

`$85:9962` は複合処理です。

```asm
85:9962  JSL $85:9C56
85:9966  JSL $80:ACE8
85:996A  JSL $84:A593
85:996E  LDA $1958
85:9971  BEQ $994A
85:9973  JSR $9D64
85:9976  JSL $80:ACE8
85:997A  JSL $84:A593
85:997E  LDA $194B
85:9981  BMI $994A
85:9983  LDA $1958
85:9986  STA $1620
85:9989  LDA $194B
85:998C  BRA $99A6
```

`$1958` と `$194B` が結果値/候補値として使われ、失敗条件なら `$994A` でtaskを消します。

## 13ゴールへの影響

今回の進みは、主に以下に効きます。

- **目標1 スクリプトVMの値の流れ**：`opcode02 -> long pointer table -> trampoline -> native subhandler -> scheduler` の流れがかなり明確化
- **目標8 条件分岐ディスパッチ系**：`0x41A10 -> 0x39850` 側が固定recordではなくVM streamとして読めるようになった
- **目標9 会話・店・イベントスクリプト仕様書**：`02 32` の実体が「イベント/状態callback登録系」だと分かった
- **目標11 外部データ化**：`0x398xx` streamを命令列として再パースする準備が進んだ

## 未確定点

- `$1923` の値 `$3A/$3B/$3C/$3D/$0D` がゲーム内で何の状態名に対応するか
- `$1620` に入る `$17/$18/$19/$192A` の意味
- `$1958/$194B` の正確な役割
- `10 07 9A F0` のVM stream上での正確な切れ方

## 次に攻める場所

次は `0x39850` 以降を、9バイトrecordではなく **VM命令列** として再パースするのが良いです。

特に、`02 32` の後ろにある `10 07` は、opcode `$10` handlerで消費される可能性が高いです。
その後の `9A F0` が opcode/data/branch target のどれかを、VM dispatcherの `opcode >= $50` 系処理と合わせて確認する必要があります。
